Mandolin

slices up Gherkin to make it more versatile

Download this project as a .zip file Download this project as a tar.gz file

What is mandolin?

mandolin is a utility for slicing up Gherkin files which have been used for Behavior Driven Development.

Why?

Gherkin is great and ideally suited to BDD. However, one of its main strengths; having a simple clear editable-by-anyone format, can also present issues with scalability and maintenance when reusing the scenarios for regression testing.

By providing a way to strip out and then re-merge tables to and from separate pipe delimited text files, data, element identifiers, expected outcomes and scope can be managed in any way you like

Let's take an example loosely based on Dan North's ATM Dispenser

Account Holder checks balance
As an Account Holder
I want to check my balance at an ATM
So that I can see how much money I have when the bank is closed

Scenario Outline: Account Holder checks their balance
Given the user enters card
When the user enters their
Then I should see "Your account balance is "

Examples:
| card | pin | balance |
| 01234567 | 1234 | $200.00 |
| 01224568 | 9999 | $0.00 |

This is fine for specifying, developing and testing our feature. However, these details will only work in an environment with the accounts, PINs and balance correctly set up. How about running the same test in a different environment?

Using with different environments

By running mandolin, the table will be removed and replace with a placeholder <> and the data will be put in a pipe delimited file. Multiple files can exist for each environment.

Getting test data from a different source - even for manual testing and refactoring

Sometimes data needs to be retrieved from a database or other source. The temptation can be to incorporate this process into a step definition. That may be OK for fully automated testing but can mean loosing transparency.

By keeping the data extraction process separate and merging the data into the feature file, the resulting assets can be accessible to technical and non technical people alike and used for manual testing, or even refactoring for product development by business analyst, tester and developer.

Testing with different scope

Sanity, regression and exhaustive testing can all use the same steps but just with different amounts of data. Feature files can end up with multiple copies of the same steps - taxing our DRY principles.

Testing with different element locators

Usually the way in which an element is located can and should stay the same. However, sometimes this isn't within your control. By putting the element locators into the tables, these can be managed separately.

Testing with different expected outcomes

Just as the balance will be different for a given account depending on the state of the system (has this account been used for any other testing), so too may other items which may need validating. Perhaps more controversially, different behaviours may also be tabulated.

Examples:
| card | pin | balance | message |
| 01234567 | 1234 | $200.00 | How much cash would you like to withdraw and would you also like a loan? |
| 01224568 | 9999 | $0.00 | Have you thought about asking your boss for a raise? |
| 01214566 | 1111 | -$200.00 | The time for excuses is over. Give us your money or we'll send someone round to persuade you. |

How do I use mandolin?

mandolin is a short script which takes arguments telling it what to do.

  • to extract tables as pipe delimited test files: mandolin.pl -e name_of_feature_file.feature
  • it will then create a new directory and place a new feature file with placeholders for tables and a sub-directory called tables which contains a series of text files each one containing a corresponding table. This may be very useful or very annoying if all you want to do is extract one or two tables. In that case, just extract the tables manually using a well-formed placeholder then perform a merge
  • to merge a feature file with one or more tables: mandolin.pl -m name_of_feature_file.feature directory_containing_the_tables neame_of the_newly_merged_feature_file
